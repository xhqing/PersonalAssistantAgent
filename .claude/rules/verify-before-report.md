# 汇报前验证规则

## 文件修改验证原则

在向用户汇报任务完成情况之前，必须执行以下验证流程：

### 1. 读取验证

- **必须重新读取**被修改的文件内容
- **不能仅凭**工具调用的返回值或上下文推断修改是否成功
- **逐行核对**关键修改点是否真正生效

### 2. 验证检查项

对于每个修改点，确认：

- [ ] 新增的代码/内容确实出现在文件中
- [ ] 修改的代码/内容已按预期变更
- [ ] 删除的代码/内容确实已被移除
- [ ] 语法正确，没有引入格式错误
- [ ] 缩进和编码风格保持一致

### 3. 验证失败处理

如果发现修改未生效或不完整：

- 重新执行修改操作
- 检查是否使用了正确的搜索和替换模式
- 如有必要，使用不同的编辑策略

### 4. 汇报要求

汇报时必须基于**实际读取到的文件内容**，而不是：

- 工具调用的成功返回状态
- 之前编辑操作中的预期内容
- 上下文中的修改计划

**错误示例**：
> "我已经将第 45 行的 `oldValue` 改成了 `newValue`"

**正确示例**：
> "验证完成，[config.ts](file:///path/to/config.ts#L45) 第 45 行现在是 `const threshold = newValue;`，修改已生效"

### 5. 适用场景

此规则适用于：

- 文件编辑（Edit）后的汇报
- 文件创建（Write）后的验证
- 文件删除（Delete）后的确认
- 任何涉及文件内容变更的操作
